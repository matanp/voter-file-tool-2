# 1.1 Committee Term Model

**Status:** Open
**Roadmap:** [SRS_IMPLEMENTATION_ROADMAP.md](../SRS_IMPLEMENTATION_ROADMAP.md) §1.1
**Effort:** 3–5 days
**Depends on:** None

## Summary

Introduce a `CommitteeTerm` model to organize committee membership into two-year periods. Add `termId` FK to committee structures, backfill existing data into a default term, and provide admin UI to create/manage terms. Leaders can only submit to the active term in v1.

## Acceptance Criteria

- [ ] New `CommitteeTerm` model: `id`, `label` (e.g., "2024–2026"), `startDate`, `endDate`, `isActive`
- [ ] Add `termId` FK to `CommitteeList` (or via new `CommitteeMembership` join model — see 1.2/1.3)
- [ ] Migration to backfill existing data into a default term
- [ ] Update all committee queries to filter by active term
- [ ] Admin UI to create/manage terms (list terms, create new term, set one as active; minimal v1: no edit of existing terms, no delete)
- [ ] Ensure leaders can only submit to the active term (v1)

## Implementation Notes

- **Schema:** See [SRS_DATA_MODEL_CHANGES.md](../SRS_DATA_MODEL_CHANGES.md) — CommitteeTerm model
- **Migration:** Per SRS_DATA_MODEL_CHANGES §5.3 (steps 1–10); backfill embedded in Tier 1 items
- **UI:** SRS_UI_PLANNING_GAPS §7 — page location `/admin/terms` or sub-section of dashboard; list terms, create new term, set one as active
