# 4.8 Additional Requirements: Governance Config Admin Controls

**Status:** Open  
**Priority:** P1  
**Effort:** 3-5 days  
**Depends on:** [1.1c Committee Governance Config](1.1c-committee-governance-config.md), [2.1 Eligibility Validation](2.1-eligibility-validation.md)

## Summary

Implement in-app admin management for `CommitteeGovernanceConfig` so party, seat-capacity, AD eligibility toggle, and non-overridable reason policy are operationally configurable and validated.

## Acceptance Criteria

### Admin API

- [ ] Add admin-only read/update endpoint(s) for governance config.
- [ ] Update validation enforces:
  - `requiredPartyCode` must exist in `DropdownLists.party`.
  - `maxSeatsPerLted` must be a positive integer in configured guardrail range.
  - `requireAssemblyDistrictMatch` is boolean.
  - `nonOverridableIneligibilityReasons` values are valid enum members.
- [ ] API returns deterministic validation errors with field-level messages.

### Admin UI

- [ ] Add governance-config admin page with form controls for all required fields.
- [ ] Party selector uses authoritative dropdown list values, not free text.
- [ ] Save flow shows diff/confirmation and success/failure feedback.

### Audit + Safeguards

- [ ] Config updates emit an audit event capturing before/after values and actor.
- [ ] Application behavior (eligibility/capacity checks) reflects updated config without restart.
- [ ] Only one effective config row remains enforceable at runtime.

### Tests

- [ ] API tests cover validation success/failure for each field.
- [ ] UI test covers load/edit/save flow and error handling.
- [ ] Integration test verifies updated config changes downstream eligibility/capacity behavior.
